# 💰 AI Personal Finance Advisor

An end-to-end personal finance analytics app: upload a bank/card statement and get automatic data cleaning, expense categorization, spending analysis, next-month forecasting, anomaly detection, and plain-English recommendations in an interactive dashboard.

Built as a portfolio-grade demonstration of a full applied data science workflow — not just a model in a notebook, but a cleaned pipeline, tested code, CI, and a real UI.

**Live components:** `dashboard/app.py` (Streamlit) · `src/` (pipeline library, fully unit tested) · synthetic 2-year dataset for demoing without any real financial data.

---

## 1. Project Overview

The app takes a raw transaction export (CSV/Excel) and turns it into a financial picture in seconds:

1. **Clean** — standardize columns/dates/currency, drop invalid rows, remove near-duplicate charges.
2. **Categorize** — rule-based merchant matching, with a TF-IDF + Logistic Regression fallback for anything the rules don't recognize.
3. **Analyze** — income, expenses, savings rate, category breakdowns, top merchants, largest transactions.
4. **Predict** — next month's expenses, a naive annual projection, and expected savings, via a linear trend model with a backtested error estimate.
5. **Detect** — unusual transactions (IsolationForest) and categories that are overspending relative to their own recent trend.
6. **Explain** — all of the above distilled into short, human-readable insights.

## 2. Business Problem

Most people don't have a clear, current picture of where their money goes until they're already over budget. Manually reconciling a spreadsheet is tedious and most budgeting apps either require linking bank credentials to a third party or give generic, non-personalized advice. This project explores what a lightweight, self-hostable, upload-and-go version of that experience could look like, using only a CSV/Excel export — no bank credentials, no cloud account required.

## 3. Dataset Description

`data/synthetic_transactions.csv` is a fully synthetic ~2-year (2024–2025) single-account transaction history generated by `src/data_generation/generator.py`, designed to look like a messy real statement:

- Biweekly salary deposits with occasional bonuses
- Fixed monthly subscriptions/bills with a few days of date jitter
- Seasonally-weighted discretionary spending (e.g. more Travel in summer, more Shopping in November/December)
- The **same real-world merchant appearing under multiple raw string variants** (`"STARBUCKS #04521"`, `"SQ *STARBUCKS"`, `"SBUX 04521 SEATTLE WA"`), mirroring how real POS/statement text is inconsistent
- Injected near-duplicate charges (~1.5% of expense rows) and ~25 injected anomalies (amount spikes, unusual wire transfers)

Ground-truth `category`, `is_duplicate`, and `is_anomaly` columns are included **only** in the synthetic data, so the cleaning and modeling pipeline can be objectively evaluated. A real uploaded statement obviously won't have these — the pipeline never relies on them, it only uses them for evaluation when present.

## 4. Data Cleaning Methodology (`src/cleaning/clean.py`)

Each step is a small, independently testable function chained by `clean_transactions()`, which returns both the cleaned DataFrame and a log of what happened (shown to the user as an "upload summary"):

1. **Schema validation** — case-insensitive check for `date`/`description`/`amount`-like columns, with common aliases mapped (`memo` → `description`, `posted_date` → `date`, etc.). Raises a clear `SchemaError` if the file can't be understood.
2. **Date parsing** — coerced to `datetime`, unparsable rows dropped.
3. **Amount coercion** — strips `$ € £ ,` before converting to numeric.
4. **Currency conversion** — if a `currency` column exists, converts to USD using a fixed rate table (a live FX API is a natural swap-in for production).
5. **Invalid row removal** — drops rows with unparsable dates/amounts, blank descriptions, or zero-amount transactions.
6. **Missing value handling** — blank descriptions → `"Unknown"`, blank categories → `"Miscellaneous"`.
7. **Deduplication** — flags transactions with the same description and amount (within 1 cent) posted within 1 day of each other as duplicates and removes them, simulating catching accidental double-charges/authorization holds.

On the bundled sample data this removes 8 of 1,238 rows as near-duplicates and 0 as invalid.

## 5. Feature Engineering & Categorization (`src/categorization/categorize.py`)

Two-layer approach:

- **Rule-based first pass** — an exact/substring lookup built from the merchant catalog (`src/data_generation/merchants.py`) plus a small keyword-regex table for generic terms (`rent`, `tuition`, `payroll`, `pharmacy`, etc.). Positive-amount transactions the rules don't catch are treated as `Salary` (income sign is itself a strong feature).
- **ML fallback** — a `TfidfVectorizer(analyzer="char_wb")` + `LogisticRegression` pipeline trained on whatever the rule layer confidently labeled, used only for descriptions the rules miss. Character n-grams (not word n-grams) were chosen deliberately: statement text is full of store numbers, abbreviations, and POS prefixes, so sub-word patterns generalize better than whole-word matching. This is the "extendable AI approach" — as more transactions get rule-labeled over time, the fallback model keeps improving without anyone touching the rule table.

On the sample data, ~99% of transactions are resolved by rules and ~1% by the ML fallback.

## 6. Model Selection & Evaluation Metrics

| Task | Model | Why | Metric (on sample data) |
|---|---|---|---|
| Categorization fallback | TF-IDF (char n-grams) + Logistic Regression | Fast, interpretable coefficients, works with very little labeled data | ~99% of rows resolved by rules; fallback used as a safety net |
| Next-month expense forecast | Linear Regression over trailing 6 months | With 12–24 months of a single account's history, a transparent trend line is easier to trust/audit than a heavier model; interface is swappable for XGBoost/Prophet later | Backtested MAE ≈ $3.2k, MAPE ≈ 41% on volatile synthetic data (real accounts with steadier habits will backtest tighter) |
| Anomaly detection | IsolationForest on amount, category-relative z-score, day-of-week | Unsupervised — doesn't require labeled anomalies in real usage; category-relative z-score keeps a $500 grocery run and a $500 flight from being judged on the same scale | Precision ≈ 0.54, Recall ≈ 0.52 against the 25 injected ground-truth anomalies |
| Overspending trend | Rule-based: latest month vs. trailing 3-month average per category | Simple, explainable, no training data required | Flags categories running > 15% over their own recent baseline |

All metrics above are computed automatically in `tests/test_models.py` and displayed live in the dashboard's "Unusual Transactions" panel whenever ground-truth labels are present.

## 7. Future Improvements

- Swap the fixed FX table for a live exchange-rate API
- Replace linear-trend forecasting with SARIMA/Prophet once more per-user history exists
- Move from SQLite/flat files to Postgres for multi-user persistence
- Add authentication and per-user data isolation for a real multi-tenant deployment
- PDF export of the monthly report (currently CSV only)
- Budget goal tracking and a savings simulator (bonus features from the original spec)
- A finance-chatbot panel backed by the Anthropic API for free-form Q&A over a user's own transactions

## 8. Installation Guide

```bash
git clone <your-fork-url>
cd finance-advisor
python -m venv .venv && source .venv/bin/activate   # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

Run the test suite:

```bash
pytest tests/ --cov=src --cov-report=term-missing
```

Regenerate the synthetic dataset (optional — a copy is already committed under `data/`):

```bash
python -m src.data_generation.generator
```

## 9. Deployment Guide

**Local:**
```bash
streamlit run dashboard/app.py
```

**Streamlit Community Cloud:**
1. Push this repo to GitHub.
2. On [share.streamlit.io](https://share.streamlit.io), create a new app pointing at `dashboard/app.py` on your `main` branch.
3. Streamlit Cloud installs from `requirements.txt` automatically — no further config needed since the app defaults to the bundled sample dataset if no file is uploaded.

**CI:** every push/PR to `main` runs the full `pytest` suite with coverage across Python 3.10 and 3.11 via `.github/workflows/tests.yml`; a coverage report is uploaded as a build artifact.

## Notes on the Synthetic Data

The `category`, `is_duplicate`, and `is_anomaly` columns in `data/synthetic_transactions.csv` are **ground-truth labels used only for evaluating the pipeline** — they don't exist in real bank exports, and the cleaning/categorization/anomaly code never reads them as input, only compares its own predictions against them where available for reporting metrics.
